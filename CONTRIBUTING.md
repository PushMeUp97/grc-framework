# Contributing to GRC Framework

Thank you for your interest in contributing to the GRC Framework! This is a community-driven project, and we welcome contributions from security professionals, compliance experts, and GRC practitioners worldwide.

## 🌍 How to Contribute

### 1. Suggesting Updates or Policy Changes

- **Open an Issue**: If you have suggestions for improving existing content or identifying gaps, please open an issue describing:
  - What needs to be updated or changed
  - Why this change is important
  - Any supporting documentation or references

- **Pull Request**: If you're ready to contribute directly:
  - Fork the repository
  - Create a feature branch (`git checkout -b feature/your-suggestion`)
  - Make your changes
  - Commit with clear messages (`git commit -m "Add/Update: description"`)
  - Push to your branch (`git push origin feature/your-suggestion`)
  - Open a Pull Request with a detailed description

### 2. Content Guidelines

When contributing, please ensure:

- **Accuracy**: Information should be accurate, well-researched, and referenced
- **Clarity**: Use clear, professional language suitable for GRC practitioners
- **Structure**: Follow existing folder and file naming conventions
- **Compliance**: Content should align with established standards (SOC 2, GDPR, NIST, ISO 27001, etc.)
- **Examples**: Include real-world examples and templates where applicable

### 3. Adding New Frameworks or Best Practices

When adding new compliance frameworks:

1. Create a new folder under `compliance/` (e.g., `compliance/iso27001/`)
2. Include a `README.md` with an overview
3. Add implementation guides and templates
4. Reference relevant standards and best practices
5. Update the main README.md

### 4. Improving Existing Documentation

- Fix typos or clarify ambiguous content
- Add missing references or external resources
- Enhance templates with additional examples
- Update outdated information

## 📝 Commit Message Convention

Use clear, descriptive commit messages:

```
Add/Update/Fix: Brief description

Detailed explanation if needed:
- Point 1
- Point 2
```

**Examples:**
- `Add: SOC 2 access control policy template`
- `Update: GDPR ROPA documentation with new requirements`
- `Fix: Typo in incident response procedure`

## 🔍 Review Process

1. All contributions will be reviewed for accuracy and alignment with the project goals
2. Constructive feedback will be provided if changes are needed
3. Once approved, your contribution will be merged into the main branch

## ❓ Questions?

- Check existing Issues and PRs to avoid duplicates
- Open a new issue with the `question` label
- Reference relevant standards or documentation
